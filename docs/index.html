01 — 核心功能

從開店、上架、搜尋,到下單出貨

覆蓋電商後台 SaaS 的完整流程,每個功能都附測試。

  1. 01

    多租戶隔離

    acts_as_tenant 行級隔離,每間店只看得到自己的資料;後台靠登入定租戶,店面靠網址 slug 定租戶。

  2. 02

    並發防超賣

    結帳用悲觀鎖(SELECT … FOR UPDATE)扣庫存,並發搶購不會扣成負數,附 race condition 測試。

  3. 03

    訂單狀態機

    AASM 管理 pending → paid → shipped,付款失敗整筆 rollback、重複回呼不重複處理。

  4. 04

    背景任務鏈

    付款成功後觸發:通知店家 → 寄買家確認信 → 出貨準備的 job 鏈。

  5. 05

    商品搜尋

    pg_search 以 Postgres trigram 搜尋,支援部分比對與中文,不另外架 Elasticsearch。

  6. 06

    公開店面 + 結帳

    免登入店面,瀏覽 / 搜尋 / 下單;結帳可選 Stripe 或綠界(ECPay),付款後寄出確認信。

  7. 07

    商品圖片 + 快取

    圖片用 Active Storage 存 Tigris(S3 相容物件儲存);店面商品卡用 Russian-doll 片段快取加速。

  8. 08

    交易信件

    付款成功後用 Resend HTTP API 寄買家確認信(主機擋 SMTP,改走 API,port 443 不被擋)。

  9. 09

    REST API + 即時看板

    對外 JSON API(/api/v1,JWT + rack-attack 限流);後台訂單用 Turbo Streams + ActionCable 即時更新。

02 — 結帳金流

可插拔的金流設計

買家可選 Stripe(信用卡)或綠界(ECPay)——兩者都是「導頁付款 + webhook 驗證」的 hosted 流程,共用同一套訂單狀態機。

01

建立訂單

結帳時先建一筆 pending 訂單與交易編號,並選擇付款方式。

02

導去金流

導到 Stripe Checkout 或綠界付款頁,買家在金流端完成付款。

03

驗證 webhook

金流 server 回呼,後端驗簽章(Stripe signature / 綠界 CheckMacValue),確認沒被竄改。

04

標記付款

驗證通過才轉為 paid 並觸發通知;重複回呼不重複處理。

03 — 技術決策

依專案規模選技術,不過度設計

多租戶
acts_as_tenant 行級隔離
不用 Apartment schema 隔離:商家一多,每張表都要逐 schema 跑 migration。
搜尋
pg_search trigram
不用 Elasticsearch:這規模不需要多養一個叢集、吃記憶體。
訂單
AASM 狀態機
狀態與轉移顯式、好讀好測,勝過土砲 enum + if。
防超賣
悲觀鎖
高併發搶購最穩;樂觀鎖會大量 retry。
金流
Stripe + 綠界,可插拔
同一套訂單 / webhook 流程接兩家,買家結帳時選。
寄信
Resend HTTP API
主機把 outbound SMTP 擋了(連線逾時),改走 HTTP API。
圖片
Active Storage + Tigris
主機沒物件儲存、磁碟又會被清空 → 接外部 S3 相容儲存。
快取 / 即時
Solid Cache + Solid Cable
快取與 ActionCable 都存 Postgres,免費方案不用 Redis。
前端
Hotwire + REST
單一技術棧、開發快;不為用不到的互動多養一套 SPA。
